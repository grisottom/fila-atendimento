Quero criar um sistema de gerenciamento de fila de atendimentos que ocorrem em agências,

Os atendimentos seguem a ordem de agendamento e a ordem de chegada de pessoas nas agências.

Cada pessoa que chega numa agência para ser atendido passa por uma triagem, depois da triagem a pessoa é chamada por um painel e se dirige para uma sala para ser atendido.

O ideal a pessoa tenha agendamento feito e chegue antes do horário, a pessoa com agendamento feito tem prioridade para ser atendida em relação à quem chega sem horário agendado no atendimento chamado de expontâneo. 
A ordem de atendimento dos espontâneos segue a ordem de chegada dos espontâneos

Os atendentes ficam em salas e chamam o próximo da fila através de um painel. É no funcionamento desse painel que quero me concentrar. Um painel atende várias salas, portanto a mensagem que será vista pela pessoa indica a sala de atendimento. 

O atendimento, seja agendado ou expontâneo, é sempre para um serviço especifico, o que significa que tanto no agendamento quanto no atendimento expontâneo a pessoa precisa informar o serviço procurado.

O acesso dos usuários será controlado e gerido pelo keycloak, os tipos de usuário são 

- gerente
- atendente

Um atributo do usuário é a agência onde ele trabalha

À qualquer horário do dia o gerente da agência alterar pode alterar as permissões dos usuários da agência.

O gerente terá permissão

- admin

Os atendentes terão permissões, também gerido por keycloak

- basica
- normal
- especial

Os usuários pré cadastrados nos servicos na agencia agencia-01, no banco de dados
- gerente, pemissão admin
- atendente-01, permissão basico,
- atendente-02, permissão normal
- atendente-03, permissão especial
- atendente-04, permissões basica, normal e especial

A senha para logar serão identicos nomes dos usuários adicionando -pwd no final

Os atendentes tem capacidades distintas e são treinados para um subonjunto de serviços, 
A relação atendente vs serviços se dá através da permissão exigida para o serviço e a permissão que o usuário possui. 

Os serviços e as permissões exigidas:
- serviço-basico, permissão basica
- serviço-normal-01, pemisssão normal
- serviço-normal-02, permissão normal
- servico-especial-01, permissão especial

Quero criar 2 aplicações, 
- Painel e
- Atendimento

# Aplicacao React chamada Atendimento para papeis Atendente e Admin

## Tela de configuração para Admin
- cadastro de agência
- cadastra paineis na Agencia do usuário Admin, informa localizacão (ex. térreo)
- cadastra salas na Agencia do usuário Admin, informa localizacao (ex. fundos, ex. 2o andar) e indica paineis para a sala, minimo 1

## Tela de triagem, feita por Atendente com permissão basica ou pelo Admin
- recepção da pessoa na Agência, por CPF
- obtem o agendamento da pessoa, se existir
- gera senha aleatória, 5 caracteres ASCII, sem acento, MAIUSCULAS e NUMEROS, e insere na tabela fila de atendimentos, senha, CPF, nome, horário agendado (se existir)

## Tela de atendimento, para Atendente e Admin
- indica em que 'sala' o atendente está atendendo
- chama o próximo 
- ausenta o próximo (não apareceu na sala, vai pro fim da fila)
- inicia o atendimento 
- finaliza o atendimento

# A aplicação Atendimento tem uma API backend em Spring-Boot, chamada API-Atendimento

A comunicação entre a aplicação de Atendimento e o Painel ocorrerá de forma assíncrona através de um broker ActiveMQ Artemis:

Aplicacao Painel vs API-Painel vs Broker

Objetivo receber as mensagens destinadas a um painel

- A aplicacao Painel cliente se conecta via SSE a uma das instâncias da API-Painel
- A API-Painel cria em tempo de execução um Subscription temporário (não durável) apontando para o endereço do tópico dinâmico (ex: agencia.42.painel.7)
- O Artemis cria uma fila interna temporária atrelada a esse endereço: enderecoTopico = "agencia." + idAgencia + ".painel." + idPainel;
- A API Painel expõe metodo para publicar Atualização, metodo com parâmetros  idAgencia, idPainel, Payload
- Quando uma mensagem é publicada naquele endereço, o Artemis entrega apenas para a instância Java onde o Painel cliente está pendurado. Quando o cliente desconecta, o Artemis destrói tudo automaticamente.

Aplicacao Atendimento
- a aplicação do Atendimento chama uma API Atendimento para obter o próximo atendimento da agencia, informando a 'sala' no chamado
- a API Atendimento obtém os 'paineis' associados à 'sala' do chamado, caso não exista retorna erro
- a API Atendimento chama uma função que obtém o próximo atendimento para a Agência (usar controle de concorrência no registro). Obs, fazer um join com os serviços atendidos pelo Atendente para selecionar o próximo
- a API Atendimento, para cada painel, entrega o Payload do atendimento para API do Painel de Atualizacao que publica no Artemis (chamada REST)
- A API Atendimento marca registro como 'chamado'

O banco de dados será Postgres e preciso que as tabelas de Agendamentos estejam pré-carregadas para testes, tanto para agencia-01 quanto agencia-02, com dados de pessoas aleatórias indexados por CPF (Numérico Long) e de agendamentos, em serviços da lista acima. 

Todos os modulos serão desenvolvidos em imagens Docker e integrados em Docker-Compose para posteriormente adaptá-los para K8s

# Replay de estado ao reconectar o Painel

Quando o Painel (re)conecta via SSE, ele precisa recuperar os atendimentos ativos (CHAMANDO, EM_ATENDIMENTO) que já estavam em curso.

Fluxo:
- Ao registrar a conexão SSE, a API-Painel publica uma mensagem {agenciaId, painelId} no tópico "replay-request" do Artemis
- A API-Atendimento escuta o tópico "replay-request"
- Ao receber, consulta o banco por atendimentos ativos da agência cujas salas estejam associadas ao painel solicitado
- Republica cada atendimento ativo no tópico do painel via /api/internal/publicar
- O SSE já está escutando e entrega os eventos ao browser, sem loop
